# 面向课题组的文档资产管理与智能助理平台（开发指导方案）

## 1. 文档目的

本文用于内部开发与技术决策，不面向客户。目标是把项目边界压清楚，明确：

1. 哪些能力必须做；
2. 哪些能力可直接复用，不要自己造轮子；
3. 哪些能力第一期不要做，避免开发量失控；
4. 在 OpenClaw 作为能力层前提下，如何用最小开发量完成首期交付。

---

## 2. 项目定位

项目不是做 Git 平台，不是做在线 Office 共编，也不是做强封堵 DLP 系统。

项目本质是：

> **围绕科研团队日常 Office 文档，构建“文件级版本 + 显式流转 + 毕业交接 + 负责人审计 + OpenClaw 智能问答”的轻量业务平台。**

核心工作模式：

- 文档由 A 处理后提交一个版本；
- 系统记录版本说明；
- A 显式流转给 B；
- B 继续处理；
- 全过程保留文件级操作日志；
- 毕业时可按成员归集历史文档并完成标准化交接；
- 管理者通过审计视图或 OpenClaw 直接问结果。

## 3. 总体技术原则

### 3.1 以复用为主，业务薄层自研
不要自研完整文档平台。采用：

- **标准化底座**：对象存储、数据库、认证、备份；
- **轻量业务层**：文档登记、状态流转、版本说明、毕业交接、文件级审计；
- **能力层**：OpenClaw 负责检索、摘要、问答、外部信息整理。

### 3.2 不碰重坑
第一期明确不做：

- 在线 Office 编辑
- 多人实时协同编辑
- Office 内容级 diff
- 自研同步客户端
- 复杂 DLP 阻断
- 极细权限矩阵
- 复杂外部协作体系

### 3.3 文件与元数据分离
文档原始文件放对象存储；业务上下文放数据库。不要把“文件本身”和“管理关系”混在一起。

建议分离：

- **文件内容**：对象存储
- **文档元数据**：数据库
- **流转记录**：数据库
- **审计事件**：数据库/日志库
- **索引与语义信息**：由 OpenClaw 或其配套索引服务消费

### 3.4 OpenClaw 做能力桥，不做底座替代
OpenClaw 的角色是：

- 文档摘要与问答
- 审计结果问答
- 毕业交接辅助总结
- 外部资料获取与整理

OpenClaw 不应替代：

- 对象存储
- 关系数据库
- 基础认证
- 业务状态机
- 审计事实存储

## 5. 软件分层建议

建议按四层做：

### 5.1 存储层
职责：存放原始文件与归档文件。

建议：
- 直接复用对象存储；
- 文件命名采用内部唯一 ID；
- 对外展示名称、版本、标签走元数据表；
- 文件版本不要覆盖，采用新对象写入。

### 5.2 业务层
职责：负责项目真正需要的管理逻辑。

应包括：
- 文档登记
- 项目/课题挂接
- 文档状态
- 显式流转
- 版本说明
- 毕业交接
- 审计事件落库

### 5.3 能力层
职责：通过 OpenClaw 提供理解与问答能力。

应包括：
- 文档摘要
- 标签生成
- 项目资料概括
- 审计结果问答
- 毕业交接摘要
- 外部资料抓取与整理

### 5.4 展示层
职责：提供用户入口与负责人入口。

建议分两类视图：
- **普通成员视图**：处理文档、发起流转、查看自己相关内容；
- **负责人视图**：看课题文档状态、交接情况、审计摘要。

## 6. 哪些不要自己造轮子

以下能力应优先复用现成组件或成熟底座：

### 6.1 对象存储
不要自研文件系统，不要自研版本文件底座。直接用成熟对象存储。

### 6.2 数据库
不要自研索引或元数据文件结构。直接用标准关系数据库存：
- 文档表
- 版本表
- 流转表
- 审计表
- 用户/角色/课题关系表
- 毕业交接表

### 6.3 用户认证
不要自己从零造账号系统。第一期尽量采用现成认证方案，哪怕先简单：
- 账号密码 / 内部账户
- 角色只分：普通成员 / 课题负责人 / 平台管理员

第一期不要搞复杂权限树。

### 6.4 备份
不要自研备份机制。直接基于对象存储和数据库做标准化备份任务。

### 6.5 Web 基础框架
不要自己造前端脚手架、权限框架、API 框架的轮子。用成熟 Web 技术栈快速搭建即可。

### 6.6 AI 大模型基础调用
不要再单独搭一个重型 AI 中台。既然 OpenClaw 已作为能力层前提，就尽量围绕 OpenClaw 提供的能力做桥接。

## 7. 必须开发的核心模块

## 7.1 文档登记与目录挂接
必须开发。

需要实现：
- 上传文档后生成文档记录；
- 绑定课题/项目/目录；
- 指定当前责任人；
- 写入基础标签和状态。

这是所有流程的入口。

## 7.2 文件级版本管理
必须开发，但保持轻量。

需要实现：
- 同一文档下产生多个版本；
- 每个版本记录提交人、时间、说明；
- 支持查看历史版本列表；
- 支持回看旧版本文件。

不要追求 Office 内容级对比，只做版本级事实记录。

## 7.3 显式流转模块
必须开发，是第一核心模块。

建议最少动作：
- 提交新版本
- 转交下一责任人
- 标记处理中
- 标记待交接
- 标记已交接
- 标记定稿
- 标记归档

每次动作都要写入：
- 操作人
- 时间
- 目标责任人（如有）
- 当前状态
- 备注说明

这部分本质上是一个轻量状态机。

## 7.4 毕业交接模块
必须开发，是第二核心模块。

建议能力：
- 按成员查询其相关文档；
- 自动筛出未归档、未挂课题、仅个人名下文档；
- 生成交接清单；
- 指定接收人；
- 记录交接完成时间与确认；
- 交接后更新文档归属。

可选增强：
- 调 OpenClaw 自动生成“毕业资料摘要”。

## 7.5 文件级审计模块
必须开发/集成。

至少记录：
- 创建
- 访问/查看
- 上传
- 下载
- 替换版本
- 流转
- 删除
- 恢复
- 归档
- 管理端关键操作

审计目标不是防死，而是让后续可回看、可管理。

## 7.6 负责人审计视图
必须开发，但范围控制住。

只面向课题负责人开放，避免做成全平台复杂审计中心。

建议查询能力：
- 近一段时间有哪些文档流转；
- 哪些文档被下载过；
- 哪些文档长期无人接手；
- 某成员毕业交接是否完成；
- 哪些文档还没有归到课题/项目名下。

## 7.7 OpenClaw 接入桥
必须开发，但尽量做薄。

本质是把业务结构化数据暴露给 OpenClaw 使用，而不是让 OpenClaw 直接接管业务数据库。

建议能力：
- 文档摘要接口
- 文档元数据查询接口
- 流转历史查询接口
- 审计结果查询接口
- 毕业交接摘要接口

用户典型问法：
- “这份文档上一版是谁处理的？”
- “课题 A 最近一个月有哪些文档在流转？”
- “某毕业成员留下了哪些关键材料？”
- “请概括这个项目空间现有资料结构。”

## 8. 第一阶段建议不做的模块

以下模块第一期建议明确排除：

### 8.1 在线 Office 编辑
理由：复杂度高，收益不匹配当前协同模式。

### 8.2 多人实时协同编辑
理由：客户已明确不强求。

### 8.3 内容级差异对比
理由：Office 类文档实现成本高，首期价值有限。

### 8.4 自研同步客户端
理由：高开发量、高维护成本，还会引入失控路径。

### 8.5 强阻断 DLP
理由：当前策略是留痕优先，不是全面限制。

### 8.6 极细权限矩阵
理由：第一期保持角色简单，减少配置与维护成本。

### 8.7 外部协作共享体系
理由：目前需求核心在内部课题组管理。

### 8.8 大而全知识图谱
理由：先把文档、流转、交接、审计跑通，知识图谱后置。

## 9. 数据对象建议

第一期建议至少有以下核心数据对象：

### 9.1 文档（Document）
字段建议：
- 文档 ID
- 标题
- 所属课题/项目
- 当前责任人
- 当前状态
- 当前版本号
- 创建时间
- 是否归档

### 9.2 文档版本（DocumentVersion）
字段建议：
- 版本 ID
- 文档 ID
- 存储对象键
- 版本号
- 提交人
- 提交时间
- 版本说明
- 文件摘要（可选）

### 9.3 流转记录（FlowRecord）
字段建议：
- 记录 ID
- 文档 ID
- 来源责任人
- 目标责任人
- 状态变更前后
- 操作时间
- 备注说明

### 9.4 审计事件（AuditEvent）
字段建议：
- 事件 ID
- 文档 ID
- 用户 ID
- 动作类型
- 时间
- 终端信息
- 附加信息

### 9.5 毕业交接（GraduationHandover）
字段建议：
- 交接 ID
- 成员 ID
- 课题/项目范围
- 接收人
- 状态
- 生成时间
- 完成时间
- 摘要说明

## 10. OpenClaw 集成建议

## 10.1 OpenClaw 应看到什么
OpenClaw 应访问的是：
- 文档元数据
- 文档摘要
- 流转历史
- 审计统计结果
- 毕业交接结果

不建议默认让 OpenClaw 直接全量读取所有原始文档内容并无边界暴露。

## 10.2 OpenClaw 最适合做什么
最适合做：
- 检索增强
- 文档摘要
- 审计结果总结
- 毕业资料概括
- 外部资料搜集与整理
- 负责人问答式查询

## 10.3 OpenClaw 不适合直接承担什么
不适合直接承担：
- 真实审计事实源
- 流转状态主写入逻辑
- 权限底座
- 文件存储底座

换句话说，OpenClaw 是“解释器”和“助手”，不是“主账本”。

## 11. 开发优先级建议

### Phase 1：最小可用
必须交付：
- 文档上传与登记
- 文档版本列表
- 显式流转
- 毕业交接
- 文件级审计
- 负责人审计视图
- OpenClaw 基础问答接入
- 基础水印

### Phase 2：体验增强
可补充：
- 自动标签
- 文档摘要优化
- 流转提醒
- 资料结构整理
- 文档归档建议

### Phase 3：智能治理增强
可补充：
- 毕业交接自动总结
- 长期无人维护文档识别
- 课题资料结构分析
- 简单风险提示

---

## 12. 一句话开发策略

如果目标是**开发量尽量小**，那就坚持这条线：

> **不自研通用底座，只自研本项目不可替代的业务薄层：显式流转、毕业交接、负责人审计视图，以及 OpenClaw 的业务桥接。**

只要守住这个边界，第一期就有较大概率做到：
- 功能够用；
- 开发量可控；
- 价值点清晰；
- 后续仍有扩展空间。
