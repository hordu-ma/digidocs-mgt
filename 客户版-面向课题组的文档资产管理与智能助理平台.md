# 面向课题组的文档资产管理与智能助理平台

## 1. 建设目标

面向山东省重点实验室随机控制科研团队，建设一套以**文档流转、历史留存、过程留痕、知识沉淀**为核心的软硬件一体化平台。

平台重点不是做 Git 式代码协作，也不是做传统网盘，而是解决以下实际问题：

- 课题组成员围绕 `docx / xlsx / pptx / pdf` 等文档开展协作时，过程不清、版本混乱、交接困难；
- 文档常沉淀在个人电脑中，毕业后资料难以完整留存和接续；
- 团队负责人希望对关键文档的流转、下载、归档等动作做到**可留痕、可回看、可追责**；
- 团队希望借助智能能力，对内部文档和外部资料进行整理、提炼与问答，提高资料利用效率。

因此，本平台定位为：

> **以 DGX Spark + OpenClaw 为智能总控，以对象存储为文档底座，以统一工作站为主要入口的科研文档资产管理与智能助理平台。**

---

## 2. 平台建设思路

平台建设遵循以下原则：

### 2.1 以文档流转替代实时共编
本项目不强求多人在线同时编辑同一份 Office 文档，而是采用更符合科研团队实际工作的**流转式协同**模式：

- A 完成一部分内容后，提交一个版本；
- 通过系统显式流转给 B；
- B 在上一版本基础上继续处理；
- 每次流转都记录责任人、时间、版本说明和状态变化。

这样可以把协作过程说清楚，而不必在 Office 文档上强行追求代码式 diff/merge。

### 2.2 以留痕优先替代高摩擦封堵
安全策略以**留痕优先**为原则，不将平台设计为“处处禁止”的高摩擦系统。

平台允许正常的工作导出和发送行为，但对关键动作进行记录，并可结合水印形成约束：

- 谁访问过文档；
- 谁上传、下载、替换过版本；
- 谁将文档交接给了谁；
- 谁执行了删除、恢复、归档等关键动作。

同时，课题组负责人可以查看审计结果，做到事后可回溯、可管理。

### 2.3 以团队资产管理替代个人文件留存
文档归属优先绑定**课题/项目/团队空间**，不依赖个人终端长期保管。

尤其针对学生毕业场景，平台将历史文档统一纳入标准交接流程，保障文档持续留存和后续接手。

### 2.4 以智能助理增强平台使用效率
OpenClaw 作为能力层接入后，不仅用于外部资料获取和整理，也用于内部文档管理辅助，例如：

- 查询某课题有哪些历史文档；
- 汇总某文档的流转情况和版本情况；
- 概括某成员毕业前遗留资料；
- 对大量文档形成摘要、标签和主题整理结果。

## 3. 预期实现的核心功能

### 3.1 文档集中管理
平台统一管理团队日常科研与工作文档，包括但不限于：

- 课题申报材料
- 研究报告
- 过程记录
- 汇报 PPT
- 数据分析表格
- 归档资料

每份文档均可挂接到具体课题、项目或团队目录中，实现集中存放、统一索引和长期保留。

### 3.2 文档版本管理
平台支持文件级版本管理，能够记录：

- 版本号
- 提交时间
- 提交人
- 版本说明
- 版本间前后关系

对于 Office 文档，不强求细粒度内容对比，而是保证版本留存、责任清晰、可回退。

### 3.3 显式流转协同
平台支持围绕文档开展显式流转，典型动作包括：

- 创建草稿
- 提交新版本
- 转交下一责任人
- 标记处理中 / 待交接 / 已交接
- 定稿
- 归档

通过这种方式，可以清楚地看到文档从谁到谁、在哪个阶段、由谁继续推进。

### 3.4 毕业文档交接
针对学生毕业这一高频场景，平台可支持标准化交接流程：

- 自动归集某成员相关历史文档；
- 按课题/项目视角形成交接清单；
- 指定接收人或课题负责人；
- 记录交接动作和时间；
- 保障交接后资料仍可检索和追溯。

这能够显著降低“人走文档乱”“材料找不到”“后续接不上”的问题。

### 3.5 文件级审计留痕
平台重点提供文件级审计，不以复杂阻断为目标，而以过程清晰为目标。可记录的审计动作包括：

- 创建
- 访问
- 上传
- 下载
- 替换版本
- 流转
- 删除
- 恢复
- 归档
- 管理员变更

审计结果默认面向课题组负责人开放，便于管理者掌握团队文档流转与资产沉淀情况。

### 3.6 水印增强
对导出或查看的重要文档，可叠加动态水印，用于强化责任提示和追溯线索。

水印可包含：

- 用户身份
- 时间
- 终端信息或项目标识

### 3.7 OpenClaw 智能助理能力
OpenClaw 作为平台能力层，可为成员和负责人提供以下辅助：

#### 面向个人成员
- 查询相关历史文档
- 汇总某课题既有材料
- 提炼文档摘要
- 辅助梳理接手上下文
- 从外部网络获取资料并整理沉淀

#### 面向课题组负责人
- 查询某课题近期文档流转情况
- 查询某毕业成员遗留资料
- 查询哪些文档长期无人接手
- 通过对话获取审计结果摘要，而不必逐份翻阅文档

## 4. 建议的总体硬件架构

### 4.1 核心组成
建议采用以下硬件组合：

- **NVIDIA DGX Spark × 1**：作为智能总控节点，承载 OpenClaw 与文档智能处理能力；
- **对象存储 × 1**：作为文档统一底座，存放原始文档、历史版本、归档资料；
- **备份存储 × 1**：用于对象存储、审计记录、元数据与流程信息的备份与恢复；
- **低配置工作站 × 10**：作为团队统一工作入口，承担文档处理、上传下载、流转和查询；
- **基础网络设备与 UPS**：保证平台稳定接入与供电安全。

### 4.2 架构分工
- **DGX Spark**：负责“理解文档、整理文档、问答文档”；
- **对象存储**：负责“保存文档、保存版本、保存归档”；
- **工作站**：负责“访问平台、处理文档、发起流转”；
- **备份系统**：负责“保障历史留存与恢复能力”。

这种分层设计能够兼顾平台能力、长期留存和后续扩展性。

## 5. 平台给客户带来的直接价值

### 5.1 文档不再随个人流失
成员毕业后，文档仍按课题和项目沉淀在平台中，后续人员可以继续接手。

### 5.2 协同过程更清楚
文档从谁流转到谁、在哪个版本、处于什么状态，都有据可查。

### 5.3 管理者可以看结果，不必翻材料
负责人可以直接获取审计结果、流转摘要和资料概况，管理效率更高。

### 5.4 智能能力提升资料利用率
通过 OpenClaw，对平台内部文档与外部资料进行统一理解、整理与问答，提升团队知识复用效率。

### 5.5 控制与体验之间更平衡
平台不走高摩擦封堵路线，而是采用“正常工作路径 + 关键动作留痕 + 水印增强”的模式，更容易真正落地使用。

---

## 6. 首期建设建议范围

为保证项目可落地、风险可控，建议首期聚焦以下能力：

- 文档集中管理
- 文件级版本管理
- 显式流转协同
- 毕业交接流程
- 文件级审计留痕
- 课题组负责人审计视图
- 动态水印
- OpenClaw 对话式查询与资料整理

暂不将在线 Office 编辑、复杂 DLP 封堵、细粒度内容比对作为首期重点，以控制建设复杂度和交付风险。

---

## 7. 结语

本方案的核心不是建设一个传统文件库，而是帮助课题组建立一套**文档可留、过程可查、交接可管、知识可用**的工作平台。

在此基础上，借助 DGX Spark 与 OpenClaw，可进一步把“文件管理”提升为“文档资产管理与智能助理协同”，使平台既能服务日常工作，也能沉淀长期科研资产。
