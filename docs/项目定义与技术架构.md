# 面向课题组的文档资产管理与智能助理平台

## 项目定义与技术架构草案

## 1. 项目概述

本项目面向课题组日常科研与管理场景，建设一套以 `docx / xlsx / pptx / pdf` 等文档为核心对象的文档资产管理与智能助理平台。

平台重点不是做 Git 式代码协作，不是做传统网盘，也不是做在线 Office 共编或强封堵 DLP 系统，而是解决以下实际问题：

- 文档协作过程不清，版本混乱，责任边界不明确；
- 文档长期沉淀在个人电脑中，毕业或人员变动后难以完整接续；
- 负责人希望掌握关键文档的流转、交接、下载、归档等过程；
- 团队希望借助智能能力提升内部资料沉淀、整理与利用效率。

平台整体定位为：

> 以对象存储为文件底座，以业务平台承载版本、流转、交接和审计，以 OpenClaw 作为深度助手能力层的科研文档资产管理与智能助理平台。

---

## 2. 建设目标

平台建设目标聚焦四个方面：

- **文档可留**：文档归属绑定团队空间和课题，不依赖个人设备长期保存；
- **过程可查**：文档的上传、版本、责任人变更和流转过程可追溯；
- **交接可管**：毕业交接形成标准流程，保障资料不断档；
- **知识可用**：借助 OpenClaw 对文档进行摘要、整理、问答和建议输出。

## 4. 核心工作模式

平台采用“文件级版本 + 显式流转”的轻量协同模式，不追求实时共编。

典型工作流程如下：

1. 成员上传文档或创建文档记录；
2. 文档挂接到团队空间、课题和子目录；
3. 当前责任人处理后提交新版本；
4. 系统记录版本说明、时间和提交人；
5. 当前责任人将文档显式流转给下一责任人；
6. 系统记录状态变化、来源责任人和目标责任人；
7. 文档继续处理中、待交接、已交接、定稿或归档；
8. 全过程形成可回看的业务记录与审计日志。

## 5. 业务组织结构与角色

### 5.1 固定组织结构

平台主组织结构固定为三级：

- 团队空间
- 课题/项目
- 子目录

文档必须挂接到该三级结构中的明确位置，避免资料游离于个人目录之外。

### 5.2 角色定义

首期角色固定为三类：

- 普通成员
- 课题负责人
- 平台管理员

首期不支持复杂权限矩阵，按角色和归属范围控制访问。

## 7. 毕业交接流程

毕业交接是首期核心模块之一，采用负责人手动发起模式。

标准流程建议为：

1. 负责人选择目标成员；
2. 系统按成员、课题和目录归集相关文档；
3. 系统筛出未归档、未规范挂接、仅个人名下等候选文档；
4. 负责人确认交接范围与接收人；
5. 系统生成交接清单；
6. 接收人确认交接；
7. 更新文档归属、责任人和状态；
8. 保存交接完成记录；
9. 由 OpenClaw 生成交接摘要和资料概览建议。

---

## 8. 负责人视图定位

负责人视图首期偏重“项目管理总览”，而不是做成事件堆叠式审计中心。

建议展示以下信息：

- 各课题文档总量
- 各状态文档数量分布
- 近期流转动态
- 长期无人接手文档
- 长期未更新文档
- 毕业交接任务状态
- 下载活跃度与关键操作摘要
- OpenClaw 给出的风险提示与整理建议

---

## 9. OpenClaw 定位

OpenClaw 在本项目中定位为“深度助手”，具备建议权和解释能力，但不直接接管主业务写入。

### 9.1 OpenClaw 负责的能力

- 文档摘要生成
- 标签和主题建议
- 历史资料归类建议
- 毕业交接摘要
- 长期无人接手文档识别
- 项目资料结构概括
- 面向成员和负责人的问答式查询
- 外部资料抓取与整理

### 9.2 OpenClaw 不负责的能力

- 对象存储底座
- 关系数据库主账本
- 权限底座
- 流转状态主写入逻辑
- 审计事实主写入逻辑

### 9.3 AI 建议原则

- AI 可以生成建议和摘要；
- AI 输出必须与正式业务动作区分展示；
- 所有建议都需人工确认后才能转化为正式状态变更。

## 10. 总体技术原则

### 10.1 以复用为主，业务薄层自研

平台不自研通用底座，只自研本项目不可替代的业务薄层：

- 文档登记
- 文件级版本管理
- 显式流转状态机
- 毕业交接
- 审计留痕
- 负责人总览
- OpenClaw 业务桥接

### 10.2 文件与元数据分离

文件和业务上下文严格分离：

- 文件内容存对象存储
- 文档元数据存关系数据库
- 流转记录存关系数据库
- 审计事件存关系数据库
- AI 摘要、标签、建议作为附属数据存储

### 10.3 面向首期可交付

系统面向单课题组、约 30 人规模，首期优先追求：

- 功能闭环清楚
- 开发量可控
- 部署简单
- 后续可扩展

## 11. 推荐部署结构

首期建议按目标架构拆分设计，即使物理部署阶段可合并，也保持逻辑边界清楚。

### 11.1 核心硬件组成

本项目围绕 2 类核心硬件构建：

- `DGX Spark`
  - 负责 OpenClaw、文档摘要、标签生成、问答、AI 建议等 AI 能力层
  - 同时承载管理平台主服务（Go 后端 API、Python Worker、前端站点、反向代理）
- `群晖 DS925+`
  - 负责数据库（PostgreSQL）、文件存储（File Station API）、共享目录、备份，以及群晖原生能力与 API 能力对接

开发环境下，P14s（CUDA 支持）模拟 DGX Spark 的角色，群晖 DS925+ 通过 Tailscale 组网接入。

首期可以依托群晖承担备份：

- 数据库备份
- 文档对象备份
- 配置文件备份
- 迁移文件与运行日志归档

### 11.2 部署职责原则

- `DGX Spark` 承担 AI 能力层与管理平台主服务，是平台正式主入口和长期宿主。
- `群晖 DS925+` 不仅是 NAS 和备份设备，还作为正式基础设施节点承担数据库、文件存储和 DSM/API 能力层。
- `群晖 DS925+` 适合作为"数据与文件基础设施中心"，不承载管理平台主服务。

### 11.3 DGX Spark 平台与 AI 能力层

承载：

- 管理平台主服务（Go 后端 API、Python Worker）
- 前端站点与反向代理
- OpenClaw
- 文档摘要、标签和问答能力
- AI 建议生成
- 外部资料抓取整理

### 11.4 群晖 DS925+ 基础设施与 API 能力层

承载：

- PostgreSQL 数据库运行节点
- 文件存储统一落点（File Station API）
- 群晖共享目录
- 原始文档、历史版本文件、归档文件与相关附件
- DSM / File Station API 对接
- 群晖原生文件、共享、权限、搜索、快照、同步等能力

基于 Synology DS925+ 官方资料，这个型号具备以下与本项目直接相关的能力：

- `4` 盘位，可通过 `DX525` 扩展到 `9` 盘位
- 最大内存 `32 GB`
- 双 `2.5GbE`
- `2` 个 M.2 NVMe 槽位，可用于缓存或存储池
- 支持 `Btrfs` 与 `ext4`
- 支持 `SMB / NFS / FTP / WebDAV / Rsync`
- 支持共享目录、版本回滚、快照、异地复制和多站点同步
- 支持 `Container Manager`
- 支持 `Virtual Machine Manager`

对本项目的直接意义是：

- 群晖不只是备份设备，而是可承担“数据库 + 对象存储 + NAS API 能力”的正式节点
- 可以利用 DSM 原生共享目录、权限、版本回滚、快照和复制能力提升文档资产可靠性
- 可以通过 DSM 与 File Station API 让平台复用群晖现成文件能力，而不是完全自造文件管理逻辑
- 若客户现场需要，也可在群晖侧运行部分容器化辅助服务
- 但群晖的最合适定位仍然是“数据与文件基础设施节点”，不是“管理平台主服务宿主”
- 管理平台主服务部署在 DGX Spark 上

### 11.5 备份策略

首期可优先依托群晖承担备份落点，不强制再增加独立备份主机，但必须明确以下备份对象：

- PostgreSQL 数据库定期备份
- 文档对象存储备份
- 审计与流程数据备份
- 环境配置和迁移文件备份

### 11.6 工作站入口

承载：

- 用户访问平台
- 文档上传下载
- 文档编辑
- 流转与查询

## 12. 技术栈建议

### 12.1 后端

- `Go 1.26`（主业务后端，标准库 net/http + database/sql）
- `Python 3.12`（Worker，承接 AI 与文档处理异步任务）
- `PostgreSQL 17`

选择理由：

- Go 适合构建高性能、类型安全的主业务 API；
- Python Worker 方便桥接 OpenClaw 等 AI 能力层；
- 标准库优先，依赖少，部署简单。

### 12.2 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Element Plus`

选择理由：

- 适合中后台项目；
- 对成员端和负责人端共享一套界面结构较友好；
- 首期不需要 SSR，引入 Vite + Vue 3 更轻。

### 12.3 数据库

- `PostgreSQL`

选择理由：

- 非常适合 `Document / Version / Flow / Audit / Handover` 这类关系型业务；
- 支持稳定的事务和查询；
- 后续扩展余地充足。

### 12.4 文件存储

- `群晖 DS925+ File Station API`

选择理由：

- 群晖 DS925+ 本身已是基础设施节点，直接复用其 File Station API 避免重复造轮子；
- 原生提供目录管理、文件搜索、共享链接、权限检查、版本快照等能力，无需在业务层重新实现；
- DSM Web API 提供完整的 HTTP 接口（SYNO.FileStation.*），Go 后端可直接 HTTP 调用；
- 统一文件落点在群晖共享目录 `DigiDocs`，文件组织结构与业务层 objectKey 一致；
- 无需额外部署 MinIO 等独立组件，降低运维复杂度。

### 12.5 异步任务

- Go 内存队列 + Python Worker HTTP 轮询

机制：

- Go 后端内置轻量任务队列，Python Worker 通过 HTTP 轮询消费任务；
- 适合承载文档摘要、标签提取、水印处理、交接摘要和定时巡检等后台任务；
- 避免大文件处理或 AI 调用阻塞主请求链路；
- 首期不引入 Redis，后续可按需升级为持久化队列。

### 12.6 部署方式

- `Docker Compose`
- `Nginx`

选择理由：

- 对 30 人规模足够；
- 易于首期交付和维护；
- 后续可以平滑演进。

## 23. 业务规则细则

### 23.1 状态流转规则

首期状态机采用固定规则，不支持项目级配置。

- `草稿 -> 处理中`
  - 触发条件：文档创建完成或首个版本上传完成后，由当前责任人开始处理。
- `处理中 -> 待交接`
  - 触发条件：当前责任人准备转交下一责任人。
- `待交接 -> 已交接`
  - 触发条件：目标责任人确认接收。
- `已交接 -> 处理中`
  - 触发条件：新责任人开始继续处理。
- `处理中 -> 定稿`
  - 触发条件：负责人或当前责任人确认文档已完成内容处理。
- `定稿 -> 已归档`
  - 触发条件：负责人或管理员执行归档。
- `已归档 -> 处理中`
  - 触发条件：负责人或管理员显式恢复归档文档。

首期不允许以下跳转：

- `草稿 -> 已归档`
- `草稿 -> 已交接`
- `待交接 -> 定稿`
- `已归档 -> 已交接`

### 23.2 角色动作规则

- 普通成员可执行：
  - 上传文档
  - 提交新版本
  - 查看自己可访问的文档
  - 发起转交
  - 接收交接
  - 查看与自己相关的 AI 建议
- 课题负责人可执行：
  - 查看所属课题全部文档
  - 发起毕业交接
  - 执行定稿、归档、恢复
  - 查看审计摘要和总览
  - 确认或忽略 AI 建议
- 平台管理员可执行：
  - 用户和课题基础配置
  - 全局审计查看
  - 全局恢复和维护操作

### 23.3 删除、恢复、归档规则

- 删除默认定义为“软删除”，不直接物理删除对象存储文件。
- 文档删除后：
  - 前端普通列表默认不再展示；
  - 审计记录保留；
  - 负责人和管理员可在回收区或管理视图中恢复。
- 归档不等于删除。
  - 归档后文档进入只读管理状态；
  - 仍可查看历史版本和审计信息；
  - 不允许普通成员继续直接提交新版本，除非先恢复到 `处理中`。

### 23.4 毕业交接完成后的自动更新

交接完成后，系统必须自动更新：

- 文档当前责任人
- 文档所属课题或目录归属信息（如负责人已调整）
- 文档状态
- 交接完成时间
- 交接审计事件
- 交接摘要可见性

## 25. 文件处理策略

### 25.1 首期支持范围

首期默认支持以下文件类型上传与版本管理：

- `pdf`
- `docx`
- `xlsx`
- `pptx`

其他类型首期可允许上传，但不承诺预览、摘要和水印能力。

### 25.2 预览策略

- `pdf`
  - 首期支持在线预览；
  - 支持导出或查看水印增强。
- `docx / xlsx / pptx`
  - 首期不做在线编辑；
  - 预览能力作为可选增强；
  - 默认保证上传、下载、版本管理和基础元数据展示。

### 25.3 摘要策略

- 默认通过异步任务生成摘要；
- 有可抽取文本时，优先基于抽取文本生成摘要；
- 文件无法抽取文本时，保留“摘要失败”状态，不阻塞主业务流程；
- 摘要失败可由用户或后台重试。

### 25.4 水印策略

- 首期重点支持 PDF 查看和导出水印；
- 水印内容包含：
  - 用户身份
  - 时间
  - 项目标识
  - 可选终端/IP 信息
- Office 原生文件水印增强暂不作为首期必交付项。

## 30. 群晖 NAS 集成预留

客户场景明确需要大量利用群晖自带能力，因此首期开发应把群晖集成作为正式基础设施边界，而不是后补插件。

这里的定位不是“把群晖只当备份盘”，而是把 `群晖 DS925+` 作为本项目的正式节点之一，承担：

- 数据库节点
- 对象存储/文件节点
- DSM / File Station API 能力节点
- 共享目录、快照、复制、同步等原生能力节点

### 30.1 集成原则

- 平台需要优先适配群晖 DS925+ 的能力，而不是默认只有独立对象存储；
- 存储与文件能力统一经由 `Synology adapter` 或统一存储适配层接入；
- 业务层不得直接散落调用群晖 API，统一经由存储适配层和集成服务调用。

### 30.2 建议优先对接的群晖能力

- DSM 登录与会话管理
- File Station 文件浏览与目录操作
- File Station 文件上传、下载、复制、移动、删除
- File Station 共享链接能力
- 权限检查
- 搜索能力
- 后台任务状态查询
- 共享目录能力
- 文件分享链接能力
- 版本回滚/快照相关能力
- 多站点同步与复制能力

### 30.3 DS925+ 官方能力与对本项目的意义

根据 Synology DS925+ 官方产品资料和 DSM 能力说明，这个型号具备以下和本项目高度相关的能力：

- 硬件与存储能力
  - `4` 盘位，可扩展到 `9` 盘位
  - 最大内存 `32 GB`
  - 双 `2.5GbE`
  - `2` 个 NVMe 槽位，可用于缓存或存储池
  - 支持 `Btrfs`
- 文件与共享能力
  - 支持 `SMB / NFS / FTP / WebDAV / Rsync`
  - 支持共享目录、版本回滚、快照、复制、同步
- 平台扩展能力
  - 支持 `Container Manager`
  - 支持 `Virtual Machine Manager`
  - DS925+ 在 Synology 的 Virtual DSM 兼容列表中，最大可运行 `8` 个 Virtual DSM 实例（在最大内存配置下）

对本项目的设计影响是：

- 群晖可以承担数据库和对象存储节点，而不只是备份盘
- 平台要把 DSM/File Station API 视为正式系统接口
- 文档平台应尽量复用群晖已有的共享、权限、搜索、快照与同步能力
- 如果现场需要，群晖还可以承载部分容器化辅助服务，但不建议替代 DGX Spark

### 30.4 已确认的官方 API 资料入口

- 群晖 File Station 官方 API 指南：
  - `https://global.download.synology.com/download/Document/Software/DeveloperGuide/Package/FileStation/All/enu/Synology_File_Station_API_Guide.pdf`
- DSM Web API 能力在不同 DSM 版本和已安装套件下会有差异，开发时应通过 `SYNO.API.Info` 动态探测目标 NAS 暴露的 API 与版本，再决定调用参数。

### 30.5 从官方文档可直接预留的接口能力

根据群晖官方 File Station API 指南，首期至少应为以下接口能力预留封装：

- `SYNO.API.Auth`
  - DSM 登录、获取会话
- `SYNO.FileStation.List`
  - 列目录、获取文件详情
- `SYNO.FileStation.Search`
  - 文件搜索
- `SYNO.FileStation.Upload`
  - 上传文件
- `SYNO.FileStation.Download`
  - 下载文件/目录
- `SYNO.FileStation.Sharing`
  - 创建和管理共享链接
- `SYNO.FileStation.CreateFolder`
  - 创建目录
- `SYNO.FileStation.Rename`
  - 重命名
- `SYNO.FileStation.CopyMove`
  - 复制与移动
- `SYNO.FileStation.Delete`
  - 删除
- `SYNO.FileStation.BackgroundTask`
  - 查询后台文件任务
- `SYNO.FileStation.CheckPermission`
  - 权限检查

### 30.6 对当前项目的实际落地建议

- 抽象统一 `StorageProvider` 接口：
  - `put_object`
  - `get_object`
  - `delete_object`
  - `list_path`
  - `move_path`
  - `create_share_link`
  - `check_permission`

  - `SynologyStorageProvider`（File Station API 适配器）

  - `storage_provider`

  - `storage_object_key`
  - `external_file_id`
  - `external_path`

同时建议新增 `SynologySystemProvider` 或等价系统适配层，用于：

- DSM 登录与会话管理
- 共享目录元数据读取
- 快照/复制状态查询
- DSM 用户与权限映射
- 群晖能力探测

### 30.7 群晖集成风险前置

- DSM 版本差异会导致 API 版本号和可用能力不同；
- 客户启用的套件不同，会影响可调用接口范围；
- 权限模型既有 DSM 用户体系，也可能有共享目录级权限，需要提前映射；
- 如果后续要接群晖分享链接、审计、快照管理或账户体系，需再确认客户是否允许平台复用 DSM 账号体系。

## 31. DGX Spark 与 OpenClaw 集成预留

OpenClaw 明确部署在 DGX Spark 上，因此平台在开发阶段应把它视为独立能力层服务，而不是本地 Python 模块。

### 31.1 集成原则

- 业务平台与 OpenClaw 通过明确接口通信；
- OpenClaw 只读取授权后的结构化数据和文档内容；
- OpenClaw 输出摘要、问答和建议，不直接写主业务数据库；
- 所有 AI 结果都要支持超时、失败重试和降级。

### 31.2 平台需要预留的 OpenClaw 对接能力

- 文档元数据读取接口
- 单文档内容抽取结果读取接口
- 文档版本历史读取接口
- 流转历史读取接口
- 审计统计读取接口
- 毕业交接信息读取接口
- 课题资料结构概览接口

### 31.3 OpenClaw 返回结果的标准化类型

- 文档摘要
- 标签建议
- 风险提示
- 交接摘要
- 问答结果
- 资料整理建议

建议统一返回：

- `request_id`
- `result_type`
- `content`
- `source_scope`
- `confidence` 或可选置信提示
- `generated_at`
- `expires_at`

### 31.4 DGX Spark 部署侧需要考虑的预留项

- OpenClaw 服务访问地址配置
- API 密钥或服务鉴权
- 超时和重试策略
- 并发限流
- 任务队列与异步回调
- 网络连通性与内网访问控制
- 失败时回退到“暂无 AI 结果，但业务可继续”的界面逻辑

### 31.5 建议预留的后端模块职责

- `assistant_client`
  - 负责与 DGX Spark 上的 OpenClaw 通信
- `assistant_tasks`
  - 负责异步摘要、建议生成、失败重试
- `assistant_cache`
  - 负责缓存问答和摘要结果
- `assistant_guard`
  - 负责权限、范围过滤、敏感内容边界控制

### 31.6 OpenClaw 对接的最小闭环

首期至少跑通以下闭环：

1. 新版本上传后触发摘要任务；
2. worker 调用 OpenClaw；
3. OpenClaw 返回摘要和标签建议；
4. 平台保存为 AI 附属结果；
5. 文档详情页和负责人总览页可展示；
6. 用户可确认或忽略建议；
7. 无论 AI 成功与否，文档版本和流转主流程都不受影响。
