# DigiDocs Mgt 异步任务消息契约

## 1. 目标

本文件定义 `backend-go` 与 `backend-py-worker` 之间的异步任务消息结构，用于支撑 AI 与文档处理链路。

目标：

- 统一任务类型命名；
- 统一请求 ID、关联对象和 payload 结构；
- 保证 Python Worker 只产出附属结果，不直接改主业务状态。

---

## 2. 基础结构

任务消息统一包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `request_id` | string | 全局请求 ID |
| `task_type` | string | 任务类型 |
| `related_type` | string \| null | 关联对象类型，如 `document`、`handover` |
| `related_id` | uuid \| null | 关联对象 ID |
| `payload` | object | 任务负载 |

结果消息统一包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `request_id` | string | 对应任务请求 ID |
| `status` | string | `completed` 或 `failed` |
| `output` | object | 输出结果 |
| `error_message` | string \| null | 失败原因 |

---

## 3. 当前任务类型

- `assistant.ask`
- `document.summarize`
- `handover.summarize`
- `document.extract_text`
- `assistant.generate_suggestion`

---

## 4. 职责边界

- `backend-go` 负责创建任务、投递任务、记录正式业务动作。
- `backend-go` 负责暴露 Worker 专用的受控上下文接口。
- `backend-py-worker` 负责消费任务、读取上下文、调用 OpenClaw、生成附属结果。
- Python Worker 不直接修改文档状态、责任人、归档状态、交接状态。

---

## 5. 后续要求

- 队列中间件确定后，需要补充重试、幂等、超时和死信策略。
- 当回写接口落地时，需要补充结果回写认证与签名要求。

## 6. 当前内部回写入口

当前实现中，Python Worker 结果回写目标为：

- `POST /api/v1/internal/worker-results`

要求：

- 使用 `Bearer` token 进行内部鉴权；
- 该接口仅接收附属结果，不处理主业务状态变更；
- 当前结果已经会持久化到 `assistant_requests` / `assistant_suggestions` 等附属表。

## 7. 当前内部上下文入口

当前 Worker 使用以下只读内部接口补充业务上下文，统一使用同一个 Worker shared token：

- `GET /api/v1/internal/assistant-context/projects/{project_id}`
- `GET /api/v1/internal/assistant-context/documents/{document_id}`
- `GET /api/v1/internal/assistant-context/handovers/{handover_id}`

设计原则：

- 只返回结构化业务上下文，不直连数据库；
- 默认不直接内嵌原始二进制文件；
- 当需要正文抽取时，Worker 通过内部文件下载接口读取版本原始文件；
- 当前最小正文抽取支持：`txt`、`md`、`csv`、`json`、`docx`；
- 对 `pdf`、图片 OCR、扫描件等尚未支持的格式，摘要任务必须降级为元数据级摘要，而不是伪装成全文摘要。
