# 测试覆盖率说明

本文记录仓库当前测试覆盖率口径、已修复缺口与剩余问题清单。

## 1. 统计口径

Go 后端覆盖率以 `backend-go/` 为边界，使用跨包覆盖率统计：

```bash
make coverage-go
```

该命令等价于：

```bash
cd backend-go
go test ./... -coverpkg=./... -coverprofile=coverage.out -covermode=atomic
go tool cover -func=coverage.out | tail -n 1
```

说明：

- `go test ./...` 的默认覆盖率是包内口径，适合查看每个 package 自身测试覆盖情况。
- `-coverpkg=./...` 是 Go 后端整体口径，能统计 handler 集成测试跨包触达的 service / repository / middleware 代码，更适合作为本项目 Go 后端总覆盖率口径。

## 2. 当前结果

本轮补测前：

- Go 后端默认包内总覆盖率：`14.4%`
- Go 后端跨包总覆盖率：`30.2%`

首轮补测后：

- Go 后端默认包内总覆盖率：`27.1%`
- Go 后端跨包总覆盖率：`38.8%`

Assistant 主链路补测后：

- Go 后端默认包内总覆盖率：`36.3%`
- Go 后端跨包总覆盖率：`40.9%`
- `internal/repository/memory/assistant_repository.go` 主路径已由 0% 提升到多数函数 70%+。
- `internal/service/assistant_service.go` 与 `internal/service/assistant_memory.go` 已覆盖新会话提问、既有会话提问、scope mismatch、任务排队、结果回写、显式记忆快照、建议查询/确认/忽略委托。

API 边界补测后：

- Go 后端默认包内总覆盖率：`40.0%`
- Go 后端跨包总覆盖率：`46.9%`
- `internal/transport/http/handlers` 包内覆盖率提升到 `40.2%`。
- 已覆盖完整 router 装配下的公开路由、受保护路由、admin 鉴权门禁、internal worker token 分支、Assistant 会话归档/恢复/建议忽略、数据资产文件夹/上传/下载/交接清单、代码仓库创建/查询/更新/删除与 push event 查询。

服务 / 纯函数 / router 补测后：

- Go 后端默认包内总覆盖率：`52.9%`
- Go 后端跨包总覆盖率：`54.6%`
- `internal/service` 包内覆盖率提升到 `83.7%`。
- `internal/transport/http/router` 包内覆盖率提升到 `100.0%`。
- `internal/repository/postgres` 包内覆盖率通过无数据库纯函数测试提升到 `14.1%`。
- 已覆盖 AdminService、CodeRepositoryService、DataAssetService、TaskService、AuditService、QueryService、PermissionService、迁移版本号解析、postgres flow/handover 状态机映射、Assistant payload / suggestion / scanner helper，以及 router 公开系统路由和受保护路由 401 分支。

PostgreSQL SQL mock 短期补测后：

- Go 后端默认包内总覆盖率：`65.3%`
- Go 后端跨包总覆盖率：`66.7%`
- `internal/repository/postgres` 包内覆盖率提升到 `55.8%`。
- `internal/queue/postgres` 包内覆盖率提升到 `70.5%`。
- 新增 `github.com/DATA-DOG/go-sqlmock` 作为 Go 后端测试依赖，并已同步 `vendor/`。
- 已覆盖 postgres queue pending 领取、默认 limit、空队列、payload decode 失败；postgres admin/auth/user/team-space/project/folder/flow/handover/document/version/dashboard/audit/code/data-folder/permission 仓储的主要 SQL 成功路径和关键 not found / validation / conflict 分支。

## 3. 已修复的覆盖缺口

已新增测试覆盖以下低风险基础层：

- `internal/config`：默认配置、环境变量覆盖、非法端口解析。
- `internal/bootstrap`：memory/postgres 容器装配、迁移目录发现、memory/synology 存储选择。
- `internal/app`：HTTP server 装配与 healthz 路由可达。
- `internal/queue/memory` 与 `internal/queue/noop`：发布、轮询、空队列和 noop 发布。
- `internal/storage/memory`：上传、下载、Stat、ListDir、CreateFolder、DeleteObject、CreateShareLink 与错误路径。
- `internal/shared`：文档上传白名单与数据资产文件名校验。
- `internal/transport/http/middleware`：middleware 链顺序、CORS、JSON Content-Type、RequestID、AccessLog、JWT Auth、RequireAdmin。
- `internal/transport/http/response`：成功响应、分页 meta 响应与错误响应。
- `internal/service`：TokenService 的生成、解析、签名错误、过期、payload 错误和 Bearer token 提取。
- `internal/repository/memory`：文档、版本、流转、交接、只读查询、权限、数据资产、代码仓库、Assistant 会话/请求/建议 memory 仓储主路径与关键错误路径。
- `internal/service/assistant_*`：Assistant 提问、会话 scope 校验、显式记忆装配、任务排队、结果回写与建议委托。
- `internal/service`：补齐 AdminService、CodeRepositoryService、DataAssetService、TaskService、AuditService、QueryService、PermissionService 的输入校验、权限结果、委托和错误传播主路径。
- `internal/repository/postgres`：补齐不依赖数据库的 flow / handover 状态机、审计 action 映射、UUID / actor helper、Assistant suggestion / scope / scanner helper。
- `internal/queue/postgres`：使用 SQL mock 覆盖 pending 领取、running 恢复 SQL、running 标记、空队列和坏 payload 分支。
- `internal/repository/postgres`：使用 SQL mock 覆盖 admin、auth、user query、team space、project、folder tree、flow、handover、document、version、dashboard、audit、code repository、data folder、permission 等主要 SQL 路径。
- `internal/db`：补齐迁移文件名版本号解析。
- `internal/transport/http/handlers`：基于完整 memory container 的 HTTP 集成测试已扩展到 router 基础装配、admin、internal worker、Assistant 会话管理、data asset、code repository 等 API 边界主路径与关键 4xx 分支。
- `internal/transport/http/router`：补齐公开系统路由装配、CORS / JSON middleware 链和受保护路由未登录 401 分支。

## 4. 剩余问题清单

距离 100% 覆盖率仍有以下问题需要继续处理：

1. `cmd/api/main.go` 仍为 0%。
   - 问题：当前 main 直接加载配置、创建 server 并阻塞启动，不适合单元测试。
   - 修复方向：把 main 内逻辑下沉为可注入 `run(ctx, getenv, listenAndServe)` 的函数，main 只保留 `os.Exit` 边界。

2. `internal/db` 迁移执行链路仍低覆盖。
   - 已覆盖：迁移文件名版本号解析。
   - 问题：`RunMigrations` 依赖真实 `*sql.DB`，当前没有 sqlmock 或测试数据库夹具。
   - 修复方向：引入 SQL mock 或 testcontainer PostgreSQL，覆盖迁移表创建、已执行跳过、按版本执行、SQL 失败。

3. `internal/queue/postgres` 已补到包内 `70.5%`。
   - 已覆盖：running 恢复 SQL、pending 领取、空队列、JSON payload decode 错误。
   - 剩余：Begin/Query/Update/Commit 失败后的回滚细分分支。

4. `internal/repository/postgres` 真实 SQL 访问层仍低覆盖。
   - 已覆盖：无数据库纯函数，以及 admin/auth/user/team-space/project/folder/flow/handover/document/version/dashboard/audit/code/data-folder/permission 的主要 SQL mock 路径。
   - 问题：Assistant、data asset 文件元数据、版本事务 workflow、action transaction 等复杂写链仍有较多 SQL 分支未覆盖。
   - 修复方向：继续按业务域拆分测试矩阵；如要提高真实性，再引入临时 PostgreSQL/testcontainer 覆盖迁移后的真实 schema。

5. `internal/repository/memory/assistant_repository.go` 仍有边界分支未覆盖。
   - 问题：Assistant 会话、请求、建议与显式记忆数据形状复杂，主路径已补，但分页、排序、不同任务类型、异常 payload、时间解析等边界还未全部覆盖。
   - 修复方向：继续补齐边界分支，尤其是不同 request_type 对 suggestion_type/title 的映射、无效 completed_at、project/document 双 scope 排序与过滤组合。

6. `internal/service` 已提升到较高覆盖，但仍有少量边界未覆盖。
   - 已覆盖：AdminService、CodeRepositoryService、DataAssetService、TaskService、AuditService、QueryService、PermissionService 主路径与关键校验。
   - 问题：少量服务的极端错误分支和外部命令失败分支仍未完全覆盖。
   - 修复方向：继续补充 git archive 失败、storage 同步失败、仓储错误传播等边界。

7. `internal/transport/http/handlers` 仍有细分边界未覆盖。
   - 问题：API 边界主路径已补到 admin、assistant、data asset、code repository、internal worker，但 handler 包仍存在大量解析失败、service 错误传播、权限组合和少量上下文接口分支。
   - 修复方向：继续按路由表补齐剩余 4xx/5xx、权限失败和 service error 分支，优先覆盖 user/admin/context 这类权限敏感接口。

8. `internal/transport/http/router` 已补到包内 `100.0%`。
   - 后续重点不再是 router 注册，而是 handler 剩余 4xx/5xx 分支和真实 PostgreSQL 仓储。

9. 若强制追求全后端 100%，需要定义是否包含启动胶水、真实 PostgreSQL 仓储、网络边界和 main。
   - 建议：100% 门禁应拆为分层目标，先让 pure/service/memory/handler 达到 90%+，再单独为 postgres 仓储设集成测试门禁；否则会把大量测试精力花在 main 和 SQL mock 细节上，而不是核心业务风险。
