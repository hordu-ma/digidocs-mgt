# 测试覆盖率说明

本文记录仓库当前测试覆盖率口径、已修复缺口与剩余问题清单。

## 1. 统计口径

Go 后端覆盖率以 `backend-go/` 为边界，使用跨包覆盖率统计：

```bash
make coverage-go
```

该命令等价于：

```bash
cd backend-go
go test ./... -coverpkg=./... -coverprofile=coverage.out -covermode=atomic
go tool cover -func=coverage.out | tail -n 1
```

说明：

- `go test ./...` 的默认覆盖率是包内口径，适合查看每个 package 自身测试覆盖情况。
- `-coverpkg=./...` 是 Go 后端整体口径，能统计 handler 集成测试跨包触达的 service / repository / middleware 代码，更适合作为本项目 Go 后端总覆盖率口径。

## 2. 当前结果

本轮补测前：

- Go 后端默认包内总覆盖率：`14.4%`
- Go 后端跨包总覆盖率：`30.2%`

本轮补测后：

- Go 后端默认包内总覆盖率：`27.1%`
- Go 后端跨包总覆盖率：`38.8%`

## 3. 已修复的覆盖缺口

已新增测试覆盖以下低风险基础层：

- `internal/config`：默认配置、环境变量覆盖、非法端口解析。
- `internal/bootstrap`：memory/postgres 容器装配、迁移目录发现、memory/synology 存储选择。
- `internal/app`：HTTP server 装配与 healthz 路由可达。
- `internal/queue/memory` 与 `internal/queue/noop`：发布、轮询、空队列和 noop 发布。
- `internal/storage/memory`：上传、下载、Stat、ListDir、CreateFolder、DeleteObject、CreateShareLink 与错误路径。
- `internal/shared`：文档上传白名单与数据资产文件名校验。
- `internal/transport/http/middleware`：middleware 链顺序、CORS、JSON Content-Type、RequestID、AccessLog、JWT Auth、RequireAdmin。
- `internal/transport/http/response`：成功响应、分页 meta 响应与错误响应。
- `internal/service`：TokenService 的生成、解析、签名错误、过期、payload 错误和 Bearer token 提取。
- `internal/repository/memory`：文档、版本、流转、交接、只读查询、权限、数据资产、代码仓库 memory 仓储主路径与关键错误路径。

## 4. 剩余问题清单

距离 100% 覆盖率仍有以下问题需要继续处理：

1. `cmd/api/main.go` 仍为 0%。
   - 问题：当前 main 直接加载配置、创建 server 并阻塞启动，不适合单元测试。
   - 修复方向：把 main 内逻辑下沉为可注入 `run(ctx, getenv, listenAndServe)` 的函数，main 只保留 `os.Exit` 边界。

2. `internal/db` 迁移与 PostgreSQL 打开链路仍为 0%。
   - 问题：`RunMigrations` 依赖真实 `*sql.DB`，当前没有 sqlmock 或测试数据库夹具。
   - 修复方向：引入 SQL mock 或 testcontainer PostgreSQL，覆盖迁移表创建、已执行跳过、按版本执行、SQL 失败、文件名版本解析。

3. `internal/queue/postgres` 仍为 0%。
   - 问题：任务轮询含事务、锁、超时重置和 JSON 扫描，当前没有数据库级测试。
   - 修复方向：使用 sqlmock 或临时 PostgreSQL 覆盖 running 重置、pending 领取、空队列、JSON 解析错误和事务回滚。

4. `internal/repository/postgres` 仍为 0%。
   - 问题：所有真实数据访问层均缺少 SQL 级测试，且 SQL 语句数量大。
   - 修复方向：按业务域拆分测试矩阵：auth/user、document/version、flow/handover、assistant、data asset、code repository、admin、permission、dashboard/audit。

5. `internal/repository/memory/assistant_repository.go` 仍为 0%。
   - 问题：Assistant 会话、请求、建议与显式记忆数据形状复杂，本轮尚未补。
   - 修复方向：先覆盖 memory 版创建会话、提问请求、完成请求、建议生成/确认/忽略、scope/memory_sources 解析，再用它支撑 service 层测试。

6. `internal/service` 中多项业务服务仍为 0% 或低覆盖。
   - 问题：AdminService、AssistantService、Assistant memory、CodeRepositoryService、DataAssetService、AuditQueryService 等尚缺系统测试。
   - 修复方向：用 fake repository / memory repository 补服务层输入校验、权限错误、仓储错误传播和成功路径。

7. `internal/transport/http/handlers` 仍为低覆盖。
   - 问题：现有 handler 集成测试只覆盖主业务少量端点，未覆盖 admin、assistant 会话管理、data asset、code repository、internal worker/context 等完整矩阵。
   - 修复方向：扩展 httptest 场景，按路由表覆盖 2xx、4xx、鉴权失败、解析失败和 service 错误。

8. `internal/transport/http/router` 包仍为 0%。
   - 问题：router.New 的实际路由装配主要被 app 测试间接触达，但包内默认覆盖率仍显示 0%。
   - 修复方向：新增 router 包测试，直接用完整 memory container 验证公开路由、受保护路由和 admin 路由。

9. 若强制追求全后端 100%，需要定义是否包含启动胶水、真实 PostgreSQL 仓储、网络边界和 main。
   - 建议：100% 门禁应拆为分层目标，先让 pure/service/memory/handler 达到 90%+，再单独为 postgres 仓储设集成测试门禁；否则会把大量测试精力花在 main 和 SQL mock 细节上，而不是核心业务风险。
