# 项目端到端总览

本文档面向需要整体理解 DigiDocs Mgt 的读者，说明这套系统从用户操作到文件存储、数据库、异步 AI 处理再回到页面展示的完整链路。

如果前面的三份代码说明文档分别回答的是“每一层内部怎么实现”，那本文档回答的是“整个项目作为一个系统，是怎么一起工作的”。

## 1. 项目最终在解决什么问题

DigiDocs Mgt 解决的是一个“毕业设计文档全过程管理”问题，核心目标包括：

- 把分散的毕业设计文档统一归档到团队空间和项目目录中；
- 让文档的创建、版本更新、处理流转、定稿、归档都有明确状态；
- 把学生之间的工作交接做成可追踪的正式业务对象；
- 让负责人能够看到整体进度、近期流转和风险文档；
- 在不破坏主业务账本的前提下，引入 AI 问答、文档摘要和结构化建议能力。

因此，这不是一个单纯的文件上传系统，也不是一个单纯的 AI 聊天系统，而是一套“文档主业务平台 + AI 助手层”的组合系统。

## 2. 系统分成哪几层

按照当前项目的正式口径，整套系统可以分成五个逻辑层。

### 2.1 用户交互层

对应目录：`frontend/`

职责：

- 提供登录、仪表盘、文档列表、文档详情、交接页、AI 助手页；
- 把后端能力组织成页面、表单、按钮、弹窗和时间线；
- 负责发起 API 请求并展示结果。

### 2.2 主业务应用层

对应目录：`backend-go/`

职责：

- 提供前端访问的 REST API；
- 维护认证、文档、版本、流转、交接、审计、仪表盘等主业务状态；
- 作为 AI 请求的主账本与结果归档中心；
- 提供 Worker 使用的内部上下文接口和文件下载接口。

### 2.3 异步 AI 执行层

对应目录：`backend-py-worker/`

职责：

- 轮询待处理 AI 任务；
- 拉取业务上下文与原始文档内容；
- 调用 OpenClaw 完成问答、摘要、建议生成；
- 把结果回写给 Go 主后端。

### 2.4 数据与文件基础设施层

职责：

- PostgreSQL：保存主业务数据、AI 请求记录、建议结果、流转与交接记录；
- 群晖 DSM / File Station：保存文档原始文件、版本文件和目录结构。

### 2.5 模型能力层

职责：

- OpenClaw Gateway 提供大模型访问能力；
- 真正的大模型理解、问答和结构化输出都在这一层发生；
- 但这一层不直接拥有主业务账本写权限。

## 3. 端到端部署视角

当前目标部署可以理解为三块物理承载：

- 应用层节点：DGX Spark；开发测试阶段可由 p14s 替代，负责运行前端、backend-go、backend-py-worker、OpenClaw 访问入口；
- 数据库层节点：群晖 DS925+ 通过 Container Manager 运行 PostgreSQL 容器；
- 文件层节点：群晖 DS925+ 原生 DSM / File Station 服务，负责文件存储与目录能力。

这样分层的好处是：

- 主业务应用与 AI 执行可以独立扩展；
- PostgreSQL 和文件系统都靠近群晖的数据底座；
- 文件层继续复用群晖原生能力，不强行重做文件系统。

## 4. 一次典型用户操作会怎样穿过整个系统

下面用几个最重要的业务场景说明端到端链路。

### 4.1 场景一：用户登录并进入系统

1. 用户在前端登录页输入用户名密码。
2. `frontend` 调用 `POST /api/v1/auth/login`。
3. `backend-go` 的认证 handler 调用认证 service。
4. 认证 service 通过 repository 校验用户并签发 JWT。
5. 前端把 token 写入 Pinia store 和 `localStorage`。
6. 后续页面请求统一带上 `Authorization: Bearer <token>`。

这条链路建立的是“谁在操作系统”的身份基础。

### 4.2 场景二：创建文档并上传首个版本

1. 用户在文档列表页点击“新建文档”。
2. 前端用 multipart 表单把文档元信息和文件提交给 `POST /api/v1/documents`。
3. `backend-go` 的文档 service 校验项目、目录、标题、责任人等业务规则。
4. service 调用 storage provider 上传原始文件。
5. 当前配置若是群晖模式，`SynologyStorageProvider` 会把文件写到群晖 File Station。
6. service 同时调用 repository 在 PostgreSQL 中写入：
   - 文档主记录；
   - 当前版本记录；
   - 审计事件。
7. Go API 把创建后的文档结果返回给前端。
8. 前端刷新列表或跳转详情页。

这条链路说明：文件实体在群晖，业务索引和状态在 PostgreSQL，页面展示由前端完成。

### 4.3 场景三：文档进入流转、定稿、归档

1. 用户在文档详情页触发“开始处理 / 转交 / 接收 / 定稿 / 归档”等动作。
2. 前端调用对应的 `/documents/{id}/flow/*` 接口。
3. `backend-go` 的流转 service 判断当前状态是否合法、操作者是否有权限。
4. repository 在 PostgreSQL 中写入新的流转记录并更新文档状态。
5. 审计 service 写入操作审计事件。
6. 前端重新拉取流转历史和文档详情，更新页面状态。

这条链路体现的是“显式流转”，也就是把本来口头发生的处理过程变成系统内可追踪状态。

### 4.4 场景四：创建交接单

1. 用户在交接页创建交接单。
2. 前端调用 `POST /api/v1/handovers`。
3. `backend-go` 的交接 service 创建交接主记录和交接项。
4. 数据写入 PostgreSQL，并记录审计事件。
5. 前端刷新交接列表。

交接单的意义是把“谁交给谁、交了哪些材料、当前是否确认完成”变成正式业务对象。

### 4.5 场景五：用户发起 AI 问答

1. 用户在 AI 助手页输入问题。
2. 前端调用 `POST /api/v1/assistant/ask`。
3. `backend-go` 创建一条 `assistant_requests` 记录，状态设为待处理。
4. 前端拿到 `request_id` 后开始轮询 `GET /api/v1/assistant/requests/{id}`。
5. `backend-py-worker` 通过内部轮询接口取到这条任务。
6. Worker 根据问题 scope 拉取相关项目或文档上下文。
7. Worker 把问题和上下文发给 OpenClaw。
8. OpenClaw 返回回答内容。
9. Worker 把结果回调给 `backend-go`。
10. Go 后端把结果更新回 `assistant_requests`。
11. 前端轮询到完成状态后展示答案。

这条链路说明：AI 是异步执行的，主业务系统负责排队、归档和对前端暴露状态。

### 4.6 场景六：生成文档摘要与 AI 建议

1. 用户在文档详情页触发“生成摘要”。
2. 前端调用 `POST /api/v1/assistant/documents/{id}/summarize`。
3. `backend-go` 创建一条文档摘要任务。
4. Worker 取到任务后，先通过内部接口拉取文档上下文。
5. 如果上下文里没有足够正文，Worker 会下载当前版本原始文件。
6. `document_text_extractor.py` 把 `txt/md/csv/json/docx` 等内容抽成纯文本。
7. Worker 把文档上下文、正文和提示词一起发给 OpenClaw。
8. OpenClaw 返回摘要文本和结构化建议列表。
9. Worker 把摘要结果和建议回写给 Go 后端。
10. Go 后端把摘要状态、输出和建议持久化到 PostgreSQL。
11. 前端重新拉取建议列表后，用户可确认或忽略建议。

这条链路是本项目“业务平台 + AI 助手层”组合最完整的体现。

## 5. 系统里的主账本和附属结果分别是什么

这是理解项目边界最关键的一点。

### 5.1 主账本

主账本由 `backend-go + PostgreSQL` 维护，包括：

- 用户与认证信息；
- 团队空间、项目、文档、版本；
- 文档流转状态与动作历史；
- 交接单与交接项；
- 审计事件；
- AI 请求状态；
- AI 建议的确认/忽略结果。

这些内容决定了系统的正式业务状态。

### 5.2 附属结果

附属结果主要由 `backend-py-worker + OpenClaw` 生成，包括：

- AI 问答内容；
- 文档摘要文本；
- 交接摘要文本；
- 结构化建议内容；
- 抽取到的文档正文。

这些内容可以辅助决策，但不直接替代主业务状态。

## 6. 为什么群晖和 OpenClaw 都重要，但职责完全不同

群晖提供的是：

- 文件存储；
- 目录结构；
- 文件下载与分享链接能力；
- 数据底座所在的基础设施环境。

OpenClaw 提供的是：

- 语言理解；
- 上下文问答；
- 摘要生成；
- 结构化建议输出。

因此：

- 群晖解决“文件放在哪里、怎么拿到”；
- OpenClaw 解决“拿到内容后如何理解和生成结果”；
- `backend-go` 解决“这些能力如何进入正式业务流程”；
- `backend-py-worker` 解决“AI 任务如何被异步、安全、可控地执行”。

## 7. 为什么 AI 不直接接管业务主流程

当前架构刻意保持了 AI 的边界：

- AI 结果默认是建议，不是正式动作；
- 正式状态更新仍由 Go 主后端的业务接口控制；
- 用户可以确认或忽略建议，而不是让模型直接改主数据；
- Worker 只能通过内部接口读取受控上下文，避免直接接触宿主环境和非业务记忆。

这样设计的目的，是避免“模型输出污染主账本”以及“宿主环境记忆污染业务结果”。

## 8. 三个代码目录应该怎么连起来看

如果从源码理解项目，建议按这个顺序：

1. 先看 `docs/项目定义与技术架构.md`，明确项目目标和系统边界。
2. 再看 `docs/API设计.md` 和 `docs/数据库设计.md`，理解契约。
3. 再看 `docs/frontend代码说明.md`，知道用户在页面上能做什么。
4. 再看 `docs/backend-go代码说明.md`，理解主业务状态如何流转。
5. 最后看 `docs/backend-py-worker代码说明.md`，理解 AI 任务如何被异步执行。

这样阅读时，能先理解“系统要求什么”，再理解“每一层如何配合实现”。

## 9. 端到端角度下，项目当前已经具备什么能力

截至当前代码状态，整套系统已经具备以下闭环：

- 用户登录与鉴权；
- 文档创建、版本上传、详情查看、软删除与恢复；
- 文档流转和归档；
- 交接单创建与查询；
- 审计事件和仪表盘总览；
- AI 问答异步执行；
- 文档摘要与 AI 建议生成、确认、忽略；
- 群晖文件存储适配；
- PostgreSQL 主账本持久化。

## 10. 后续扩展最可能落在哪些位置

如果后续继续演进，这几个方向最自然：

- 在 `backend-py-worker` 增强 PDF、OCR、扫描件文本抽取；
- 在 `backend-go` 增加更细粒度的权限和审批规则；
- 在前端补充更完整的建议工作台与审计视图；
- 在业务层实现显式 AI 记忆，而不是依赖模型宿主环境记忆；
- 在 OpenClaw 接入上继续做白名单 skill 复用与隔离控制。

## 11. 一句话总结整套系统

可以把 DigiDocs Mgt 理解为：

“以前端作为操作入口、Go 作为主业务中枢、PostgreSQL 作为主账本、群晖作为文件底座、Python Worker 作为异步 AI 执行器、OpenClaw 作为模型能力来源的一套毕业设计文档管理与 AI 辅助平台。”
