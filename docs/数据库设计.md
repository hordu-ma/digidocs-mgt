# 面向课题组的文档资产管理与智能助理平台

## 数据库设计草案

## 1. 设计原则

- 使用 `PostgreSQL` 作为主数据库。
- 所有主键统一使用 `UUID`。
- 所有时间字段统一使用 `timestamptz`。
- 文档主记录、版本记录、流转记录、交接记录、审计记录分表存储。
- AI 结果与主业务事实分离，不直接写入主业务状态。
- 删除采用软删除，文件本体不因业务删除立即物理清理。

## 4. 关系说明

- 一个 `team_space` 下有多个 `projects`。
- 一个 `project` 下有多级 `folders`。
- 一个 `document` 属于一个 `project`，可挂接一个 `folder`。
- 一个 `document` 有多个 `document_versions`。
- 一个 `document` 有多条 `flow_records`。
- 一个 `document` 可被多个 `graduation_handover_items` 关联，但同一交接单中只能出现一次。
- 一个 `document`、`handover`、`project` 都可被 `assistant_suggestions` 关联。
- 一个 `assistant_conversation` 绑定单一 `scope_type + scope_id`。
- 一个 `assistant_conversation` 下有多条 `assistant_conversation_messages`。
- 一个 `assistant_request` 可选归属于一个 `assistant_conversation`。

## 5. 约束与触发逻辑建议

### 5.1 应用层事务

以下操作必须在单事务中完成：

- 上传新版本 + 更新 `documents.current_version_id` + 写审计事件
- 状态流转 + 更新当前责任人/状态 + 写流转记录 + 写审计事件
- 交接完成 + 批量更新文档责任人 + 写交接状态 + 写审计事件

### 5.2 建议的数据库检查约束

- `documents.is_archived = true` 时，`current_status` 应为 `archived`
- `documents.is_deleted = true` 时，`deleted_at` 不为空
- `document_versions.version_no > 0`
- `assistant_suggestions.confidence` 范围应在 `0` 到 `1`

### 5.3 不建议用数据库触发器承载业务

状态机规则、权限校验、AI 建议确认逻辑建议放在应用层，不建议用大量触发器实现。
