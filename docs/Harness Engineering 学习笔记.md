# Harness Engineering 学习笔记

## 1. 什么是 Harness Engineering

在这个项目里，我把 Harness Engineering 理解为一套“让人和代理都稳定做对事”的工程组织方式。

它的重点不是单次把代码写出来，而是把下面几件事工程化：

- 目标是否清楚
- 边界是否清楚
- 契约是否先于实现
- 状态是否可追踪
- 环境是否可恢复
- 代理是否会在正确的上下文里工作

如果没有 harness，大模型每次都像“重新进组的实习生”。
如果 harness 做得好，大模型更像“进入标准作业面的新工程师”，会先读对文档、走对流程、用对验证方式。

## 2. 为什么这个项目很适合学习 Harness Engineering

`DigiDocs Mgt` 不是一个纯前端 demo，也不是一个单体脚本项目，而是一个有明显业务边界和外部集成边界的系统：

- 有业务主账本
- 有 AI 附属能力层
- 有对象存储和外部系统适配层
- 有固定状态机
- 有 Go / Python / Vue 多栈协作

这类项目天然需要 harness。

因为如果没有明确约束，大模型非常容易犯以下错误：

- 跳过设计文档，直接写实现
- 在业务代码里散落外部系统调用
- 把 AI 建议误当正式业务事实
- 私自增加状态、字段或接口语义
- 写完代码却不更新 README 和 TASKS

## 3. 这个仓库改造前的问题

改造前，这个项目已经有不错的“文档治理意识”，但还没有形成完整 harness：

- 已有 `AGENTS.md`
- 已有三份核心设计文档
- 已有 `README.md` 和 `TASKS.md`
- 已有 `SKILLS/` 目录

但关键缺口有三类。

### 3.1 文档存在，不等于代理可执行

仓库里的 `SKILLS/*.md` 更像项目经验说明，而不是 Codex 可发现、可触发、可安装的真实 skill。

这会导致两个结果：

- 人类能看懂，但模型不一定会稳定启用
- 技能有名字，但没有触发条件、所有权范围、验证入口

### 3.2 原则存在，不等于流程闭环

`AGENTS.md` 已经提出“先文档、后契约、再实现”，但没有把这件事操作化为：

- 会话开始先读什么
- 改代码前要确认什么
- 改完后必须更新什么
- 用什么命令验证

### 3.3 状态账本存在，但会漂移

改造前：

- `README.md` 显示项目已进入 Go 主业务迁移的真实读写补齐
- `TASKS.md` 还写着“项目骨架初始化阶段”

这说明项目有状态记录，但还没有建立“单一事实源 + 同步规则”。

## 4. 这次改造做了什么

这次改造的核心目标，是把“协作原则”升级为“可执行 harness”。

### 4.1 新增真实 skill 包

新增目录：

- `ops/codex/skills/digidocs-architecture-guard`
- `ops/codex/skills/digidocs-backend-go`
- `ops/codex/skills/digidocs-worker-python`
- `ops/codex/skills/digidocs-frontend-vue`
- `ops/codex/skills/digidocs-doc-sync`
- `ops/codex/skills/digidocs-verify`

每个 skill 都补了最重要的运行要素：

- 触发条件
- 必读文档
- 所有权范围
- 强约束
- 最小验证方式
- 完成检查

这和普通说明文档的区别在于：

- 它不只是“讲道理”
- 它直接定义代理的工作入口和退出条件

### 4.2 新增技能索引

新增：

- `ops/codex/skills/index.yaml`

它把技能目录变成一个机器可读索引。这样做的意义是：

- 团队可以看到有哪些 skill
- 后续可以写安装、检查、同步工具
- 触发词、依赖文档、验证命令不再散落

### 4.3 新增安装脚本

新增：

- `scripts/codex/install-project-skills.sh`

这个脚本的作用，是把仓库里的项目 skill 链接到 `~/.codex/skills/` 运行时目录。

这是非常关键的一步。

因为 Harness Engineering 的一个核心思想是：

> 约束必须能进入运行时，而不是只停留在仓库文字里。

### 4.4 新增环境体检脚本

新增：

- `scripts/codex/doctor.sh`

它统一检查：

- 基础命令是否存在
- 核心文档是否存在
- Codex 相关资产是否存在
- 进度文档标记是否存在

这体现了 Harness Engineering 的另一个核心点：

> 不依赖记忆，优先提供可重复执行的体检入口。

### 4.5 强化 AGENTS.md

这次对 `AGENTS.md` 的增强重点，是把原则变成执行协议：

- 增加会话启动清单
- 增加技能使用约定
- 增加状态账本同步规则
- 增加验证与交付证据要求
- 增加冲突解决规则

这让 `AGENTS.md` 从“协作文档”升级成“仓库级操作协议”。

### 4.6 第二阶段补上统一验证入口

第二阶段继续把 harness 往前推进了一步，补的是“统一验证编排”：

- 新增 `Makefile`
- 新增 `scripts/codex/check-doc-sync.sh`
- 让 `doctor.sh` 自动调用文档一致性检查
- 提供 `make verify` 作为统一聚合验证入口

这一步很重要，因为很多项目虽然有规范，但没有统一验证入口。

没有统一入口时，团队成员和代理会出现三种常见问题：

- 知道要验证，但不知道按什么顺序跑
- 只跑自己熟悉的那部分检查
- 文档约束和代码验证分离，导致“代码通过但账本漂移”

`make verify` 的价值就在于把这些动作编排成固定路径：

1. 先跑环境体检
2. 再跑文档一致性检查
3. 再跑 Go 测试
4. 再跑 Python Worker 测试
5. 最后跑前端构建

这体现了 Harness Engineering 里很重要的一条原则：

> 验证本身也应该被 harness 化，而不是靠人临场判断。

### 4.7 第三阶段补上自动化门禁与状态外显

第三阶段做的不是再写更多说明，而是把 harness 继续推进到“自动守门”：

- 增加本地 git hooks
- 增加 GitHub Actions `verify.yml`
- 增加 `report.sh`
- 增加 `smoke-local.sh`

这四件事分别解决四种常见薄弱点。

第一，本地提交门槛。

如果没有 hooks，很多规范只有在“想起来的时候”才会执行。
有了 `.githooks/pre-commit` 和 `.githooks/pre-push`，文档同步和统一验证会更早发生。

第二，远端合并门槛。

如果没有 CI，团队成员本地环境不同，最后仍然可能把问题推到主分支。
GitHub Actions 的价值，是给仓库提供一个共享且稳定的验证裁判。

第三，状态外显。

如果一个项目的健康状态只能靠记忆描述，它的 harness 成熟度仍然不高。
`report.sh` 的意义，是把当前分支、阶段、技能安装状态、hooks 状态快速打印出来。

第四，联调烟测。

`make verify` 更偏静态验证和子系统验证。
但工程项目还需要回答一个问题：

> 容器和服务真正联起来后，最小链路有没有活着？

`smoke-local.sh` 现在提供的是第一版骨架，它先解决“有没有统一入口”，后续再逐步细化到关键业务闭环。

## 5. 这次改造体现了哪些 Harness Engineering 原则

### 5.1 文档优先

先有设计文档，再有实现。

在这个项目里，文档优先不是写写 README，而是明确三层契约：

- 架构契约
- 数据契约
- API 契约

### 5.2 环境优先

一个好的工程环境，不只是能跑代码，还要能稳定引导代理做正确的事。

所以需要：

- skill
- index
- install script
- doctor script
- session startup checklist

### 5.3 状态优先

Harness Engineering 不允许“大家都觉得项目在推进”，但没有一个稳定的阶段账本。

状态必须：

- 有单一事实源
- 有同步规则
- 有更新责任

### 5.4 验证优先

“我觉得差不多”不算完成。

完成必须带最小验证证据，比如：

- `go test ./...`
- `uv run pytest -q`
- `npm run build`
- `./scripts/codex/doctor.sh`

### 5.5 边界优先

在这个项目里，最关键的边界有三条：

- 群晖调用不能散落在业务代码中
- OpenClaw 不能成为主业务账本
- AI 建议不能伪装成正式动作

Harness 的价值，就是把这些边界从“口头共识”变成“代理必须遵守的执行条件”。

## 6. 我应该如何在 Obsidian 里理解这套改造

可以把这套体系理解成四层：

### 6.1 第一层：目标层

- `项目定义与技术架构.md`

回答“这个系统到底要解决什么问题”。

### 6.2 第二层：契约层

- `数据库设计.md`
- `API设计.md`

回答“系统里的事实和接口应该长什么样”。

### 6.3 第三层：执行层

- `AGENTS.md`
- `ops/codex/skills/*`

回答“人和代理应该按什么顺序工作”。

### 6.4 第四层：运行层

- `scripts/codex/install-project-skills.sh`
- `scripts/codex/doctor.sh`

回答“这些约束如何真正进入运行时并被检查”。

## 7. 以后继续改造时的优先顺序

后续还可以继续增强，但建议按下面顺序推进。

1. 先把项目 skill 真正安装到 `~/.codex/skills/`
2. 再补一个 `make verify` 或统一验证入口
3. 再做 README / TASKS 自动同步或检查脚本
4. 再考虑把 skill index 接入更自动的检查流程

不要一开始就追求复杂自动化，先把最关键的行为路径稳定下来。

## 8. 判断一个项目是否有良好 harness 的简表

你以后看一个项目，可以直接问这 8 个问题：

1. 代理开始工作前，知道先读什么吗？
2. 设计文档、数据库设计、API 契约之间有明确优先级吗？
3. 项目边界是否写成了硬约束？
4. 技能是“能触发的运行时实体”还是“只是一些说明文档”？
5. 环境诊断是否有统一入口？
6. 项目阶段和任务状态是否有主账本？
7. 代码完成后是否要求验证证据？
8. 多人或多轮接力时，下一位是否能无损接手？

如果这 8 个问题大多回答不上来，这个项目的 harness 往往还不够成熟。

## 9. 对我最重要的记忆点

最后把最值得记住的一句话写下来：

> Harness Engineering 不是让模型更聪明，而是让环境更不依赖模型临场发挥。

再简化一点：

> 好的 harness，会把“应该怎么做”提前写进环境，而不是留到每次对话里临时解释。
