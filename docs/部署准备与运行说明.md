# DigiDocs Mgt 部署准备与运行说明

## 1. 目标

本文档描述两件事：

- 本项目仓库内已经具备的部署资产；
- 当前推荐的目标部署分层与启动顺序。

外部前提：

- OpenClaw Gateway 已在目标机器上安装并可通过 HTTP API 访问；
- Synology DSM / File Station API 已准备好服务账号与共享目录；
- PostgreSQL 已确定运行位置与备份策略。

## 2. 目标部署分层

当前推荐按三层理解正式部署：

1. 应用层
  - 机器：`DGX Spark`
  - 开发测试替代：`P14s`
  - 承载：`frontend`、`backend-go`、`backend-py-worker`、反向代理、OpenClaw Gateway
2. 数据库层
  - 机器：`群晖 DS925+`
  - 承载：`PostgreSQL`
  - 推荐方式：使用群晖 `Container Manager` 运行数据库容器
3. 文件层
  - 机器：`群晖 DS925+`
  - 承载：DSM / File Station / 共享目录 / 快照 / 权限 / 备份等原生能力

说明：

- 群晖文件层优先复用 DSM 原生服务，不建议为了形式统一再额外容器化一套文件服务；
- 仓库根目录现有 [docker-compose.yml](../docker-compose.yml) 主要用于单机开发联调，不等同于最终双节点部署形态。

## 3. 仓库内部署资产

- 根目录 [docker-compose.yml](../docker-compose.yml)
  - 启动 `postgres`、`backend-go`、`backend-py-worker`、`frontend`
- 根目录 [.env.example](../.env.example)
  - 本地开发模板
- 根目录 [.env.production.example](../.env.production.example)
  - 部署模板
- 前端镜像 [frontend/Dockerfile](../frontend/Dockerfile)
- 前端反向代理 [frontend/nginx/default.conf](../frontend/nginx/default.conf)

## 4. 推荐启动方式

### 4.1 本地单机联调

1. 复制生产模板：

```bash
cp .env.production.example .env
```

2. 按实际环境修改 `.env`

重点字段：

- `DATABASE_URL`
- `JWT_SECRET`
- `WORKER_CALLBACK_TOKEN`
- `CALLBACK_TOKEN`
- `OPENCLAW_BASE_URL`
- `OPENCLAW_API_KEY`
- `STORAGE_BACKEND`
- `SYNOLOGY_*`

要求：

- `CALLBACK_TOKEN` 应与 `WORKER_CALLBACK_TOKEN` 保持一致；
- `CALLBACK_BASE_URL` 在 compose 网络内应保持为 `http://backend-go:8080`；
- 若 OpenClaw 运行在宿主机且 Worker 在容器内，仓库当前 compose 已为 Linux / p14s 注入 `host.docker.internal:host-gateway`，可统一使用 `http://host.docker.internal:18789`。
- macOS / Docker Desktop 也可继续使用 `host.docker.internal` 访问宿主机 OpenClaw 端点；
- 本地联调时可继续使用 compose 内置 `postgres`；正式双节点部署时应改为群晖侧数据库地址。

3. 启动前先验证 OpenClaw 兼容端点：

```bash
curl -fsS http://127.0.0.1:18789/v1/models
```

说明：

- `openclaw status` 或 `openclaw gateway health` 只能证明 Gateway/Control UI 存活；
- 只有 `GET /v1/models` 与 `POST /v1/chat/completions` 可用，当前仓库 Worker 才能真正调用 AI 能力。

4. 启动服务：

```bash
docker compose --profile app up -d --build --remove-orphans
```

5. 查看状态：

```bash
docker compose ps
make status
```

6. 执行严格烟测：

```bash
STRICT_SMOKE=1 make smoke
```

说明：

- 该命令应作为本地单机联调完成后的最小验收门槛；
- 当前 smoke 已覆盖登录、文档列表、Dashboard、交接列表、审计查询、版本上传/下载/预览、Assistant 问答完成态；
- 若本机同时配置了 `STORAGE_BACKEND=synology` 或显式传入 `RUN_SYNOLOGY_PREFLIGHT=1`，还会追加 DSM / File Station API 前置验收；
- 当前 Assistant 轮询窗口默认按 `20` 次、每次 `2` 秒执行；若现场大模型响应更慢，可通过 `ASSISTANT_SMOKE_POLL_ATTEMPTS`、`ASSISTANT_SMOKE_POLL_INTERVAL` 临时放宽，但不建议静默跳过。

### 4.2 目标双节点部署

推荐拆成两组：

1. `DGX Spark / P14s` 应用层
  - 启动 `frontend`、`backend-go`、`backend-py-worker`
  - OpenClaw Gateway 可运行在同机容器或同机宿主环境
2. `群晖 DS925+` 数据库层
  - 使用 `Container Manager` 启动 `PostgreSQL` 容器
3. `群晖 DS925+` 文件层
  - 直接使用 DSM / File Station / 共享目录原生服务

关键配置要求：

- `DATABASE_URL` 指向群晖侧 PostgreSQL 容器的固定地址；
- `SYNOLOGY_HOST` 指向群晖 DSM 固定地址；
- `STORAGE_BACKEND` 设为 `synology`；
- 若目标 DSM 仍使用自签名证书，可临时设置 `SYNOLOGY_INSECURE_SKIP_VERIFY=true`；
- `CALLBACK_BASE_URL` 若 Worker 与 Go API 同机部署，可继续使用应用层内部地址；
- 不要求依赖 Tailscale，只要求应用层机器可以稳定访问群晖固定地址与相应端口。

## 5. 访问入口

- 前端：`http://localhost:${FRONTEND_PORT}`
- Go API：`http://localhost:${BACKEND_GO_PORT}/healthz`
- PostgreSQL：`localhost:${POSTGRES_PORT}`

## 5.1 最终 TLS / 反向代理 / 防火墙口径

推荐正式部署按下面口径收口：

1. TLS 终止层
   - 放在应用层主机（`DGX Spark / P14s`）入口反向代理；
   - 推荐由宿主机 Nginx / Caddy 统一监听 `443`；
   - `80` 只保留跳转到 `443`。
2. 反向代理转发
   - `/` -> `frontend` 容器（静态资源与前端路由）；
   - `/api/v1/` -> `backend-go:8080`；
   - `backend-py-worker` 不直接暴露给浏览器或外部网络；
   - OpenClaw 兼容 HTTP 端点仅保留给本机/内网服务访问，不作为公网入口。
3. 应用层防火墙
   - 对用户侧只开放 `80/443`；
   - `22` 仅对运维管理网段开放；
   - `18080/18081` 这类容器调试端口不作为正式外部入口长期暴露。
4. 群晖侧防火墙
   - `5001`（DSM / File Station HTTPS）仅对应用层主机开放；
   - `5432`（PostgreSQL）仅对应用层主机开放；
   - DSM 管理入口与 SSH 仅对管理员网段开放，不对普通用户网段开放。
5. 网络约束
   - 用户浏览器永远只访问应用层反向代理，不直接访问群晖；
   - 应用层主机负责访问群晖 DSM / PostgreSQL；
   - 当前仓库的 compose 应用网络固定为 `172.29.0.0/24`，避免 Docker 每次重建时漂移到新的桥接子网；
   - 若继续使用 Tailscale，需确保 `table 52` 至少对 `172.17.0.0/16`、`172.29.0.0/24`、`192.168.1.0/24` 写入 `throw` 路由，避免覆盖 Docker 网桥与本机局域网回流；
   - `./scripts/codex/install-persistent-routing.sh` 会安装对应的 `tailscaled` 持久化修复脚本，并保留额外例外网段（如 `8.152.204.76/32`）的注入口径。

## 6.1.2 持久化路由修正

当前仓库已提供 Linux 宿主机持久化安装入口：

```bash
./scripts/codex/install-persistent-routing.sh
```

该安装动作会完成：

1. 把仓库内 `ops/systemd/digidocs-route-fix.sh` 安装到 `/usr/local/bin/digidocs-route-fix.sh`
2. 把 `ops/systemd/tailscaled-digidocs-routing.conf` 安装到 `/etc/systemd/system/tailscaled.service.d/digidocs-routing.conf`
3. `systemctl daemon-reload`
4. 立即执行一次路由修正脚本，确保当前会话生效

安装后的持久化机制：

- 每次 `tailscaled` 启动或重启后，都会自动执行 `/usr/local/bin/digidocs-route-fix.sh`
- 目标是始终保持以下三条规则存在于 `table 52`：
  - `throw 172.17.0.0/16`
  - `throw 172.18.0.0/16`
  - `throw 192.168.1.0/24`

建议安装后执行：

```bash
systemctl show tailscaled --property=DropInPaths --no-pager
ip route show table 52
sudo systemctl restart tailscaled
ip route show table 52
```

若重启 `tailscaled` 后上述三条 `throw` 规则仍在，则说明持久化配置已生效。

补充说明：

- 当前脚本会在短时间内重复回写几轮 `throw` 路由，以覆盖 `tailscaled` 启动后延迟补写 `table 52` 的窗口；
- 已在当前开发机验证通过，无需再手工执行 `ip route replace throw ... table 52`。

## 6.1.3 前端 18080 / 后端 18081 访问超时排查

若通过 RustDesk 登录远程桌面后，浏览器访问 `http://127.0.0.1:18080` 或 `http://127.0.0.1:18081/healthz` 超时，但容器本身仍在运行，优先按本节排查宿主机路由表。

典型现象：

1. `docker compose ps` 显示 `frontend`、`backend-go` 等容器均为 `Up`；
2. `ss -ltnp` 能看到宿主机正在监听 `0.0.0.0:18080` / `0.0.0.0:18081`；
3. 容器内访问前端或后端正常；
4. 宿主机或 RustDesk 桌面浏览器访问 `127.0.0.1:18080` 超时。

优先检查 `tailscaled` 的 `table 52` 是否重新接管了 Docker 网桥或局域网网段：

```bash
ip route show table 52 | grep -E 'throw 172\.17\.0\.0/16|throw 172\.18\.0\.0/16|throw 192\.168\.1\.0/24'
```

期望能看到以下三条规则：

```text
throw 172.17.0.0/16
throw 172.18.0.0/16
throw 192.168.1.0/24
```

若没有输出，或看到 `172.17.0.0/16` / `172.18.0.0/16` 继续指向 `tailscale0`，则说明 Tailscale 路由表重新覆盖了 Docker / 局域网回流规则。此时先执行一次已安装的修复脚本：

```bash
sudo /usr/local/bin/digidocs-route-fix.sh
```

若当前机器尚未安装到 `/usr/local/bin`，可在仓库根目录临时执行：

```bash
sudo DIGIDOCS_ROUTE_FIX_ATTEMPTS=1 ops/systemd/digidocs-route-fix.sh
```

修复后验证：

```bash
curl -I http://127.0.0.1:18080
curl -I http://127.0.0.1:18081/healthz
ip route show table 52
```

如果问题在 `tailscaled` 重启、网络切换、Docker 网络重建后反复出现，先确认持久化安装仍在：

```bash
systemctl show tailscaled --property=DropInPaths --no-pager
sudo ./scripts/codex/install-persistent-routing.sh
```

当前持久化方案通过 `tailscaled` 的 systemd drop-in 在服务启动或重启后自动执行修复脚本。若未来观察到 `tailscaled` 在运行中动态刷新 `table 52` 并再次覆盖这些 `throw` 规则，可再补一个 systemd timer，周期性执行 `/usr/local/bin/digidocs-route-fix.sh`，作为运行期兜底。

## 6.1.4 同一局域网用户访问方式

本地 compose 联调时，前端容器默认暴露在宿主机 `18080`，后端 Go API 默认暴露在宿主机 `18081`。同一局域网内其他用户访问时，应使用宿主机局域网 IP，而不是 `127.0.0.1`。

当前开发机示例：

```text
http://192.168.1.142:18080
```

使用前确认：

```bash
hostname -I
docker compose ps
ss -ltnp | grep -E ':18080|:18081'
curl -I http://127.0.0.1:18080
curl -I http://127.0.0.1:18081/healthz
```

说明：

- `127.0.0.1:18080` 只代表访问者自己机器上的端口，不能给同网段其他用户使用；
- 同网段用户应访问 `http://<宿主机局域网IP>:18080`；
- 若目标机器开启防火墙，需要允许局域网访问 TCP `18080`，后端 `18081` 建议只在联调时暴露；
- 正式部署仍建议通过反向代理和域名入口访问，不长期暴露 `18080/18081` 作为生产入口。

  - 确认 `assistant.ask -> completed` AI 闭环通过

6. 做人工联调
  - 新建一个测试文档
  - 上传第二版文件
  - 下载和预览新版本
  - 触发一次文档摘要
  - 在前端回看 Assistant 历史和结果详情

7. 记录联调结果
  - 成功项
  - 失败项
  - 环境参数
  - 是否需要长期保留 `SYNOLOGY_INSECURE_SKIP_VERIFY`
  - 是否需要补 DSM 权限或共享目录策略

## 8. 回滚方式

如果真实群晖联调失败，优先回滚到 memory 存储，避免阻塞其他业务开发：

1. 把 `STORAGE_BACKEND` 改回 `memory`。
2. 清空或注释 `SYNOLOGY_*` 环境变量。
3. 重启 `backend-go` 服务或重新执行 `docker compose --profile app up -d backend-go`。
4. 重新执行 `make smoke`，确认文件链路与 AI 闭环恢复。

## 9. 当前已知限制

- 如果 `curl http://127.0.0.1:18789/v1/models` 失败，则本轮只能完成非 AI 链路烟测，AI 任务会因为宿主机未提供兼容 HTTP 端点而失败；
- 图片 OCR 与扫描 PDF OCR 依赖 Worker 主机安装 `tesseract`；
- 前端构建仍有 bundle size warning，但不阻塞部署；
- OpenClaw HTTP 端点是否启用，仍取决于目标机器上的 Gateway 配置；
- 群晖 DSM 若使用自签名证书，正式运行前仍需补可信证书或证书信任链；
- 不同 DSM / File Station 版本对 `SYNO.API.Auth` 的入口路径可能不同；当前环境已确认需要 `webapi/entry.cgi`，因此群晖登录实现与 smoke 脚本需避免写死 `webapi/auth.cgi`；
- 若运行镜像未及时重建，可能出现“仓库 migrations / 代码已更新，但容器内二进制与 SQL 文件仍为旧版本”的偏差，部署验收前需显式确认镜像内容与 `schema_migrations` 一致。
