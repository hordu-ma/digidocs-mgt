# DigiDocs Mgt 部署准备与运行说明

## 1. 目标

本文档只描述“本项目仓库内已经具备的部署资产”和“启动顺序”。

外部前提：

- OpenClaw Gateway 已在目标机器上安装并可通过 HTTP API 访问；
- Synology DSM / File Station API 已准备好服务账号与共享目录；
- PostgreSQL 已确定运行位置与备份策略。

## 2. 仓库内部署资产

- 根目录 [docker-compose.yml](/Users/liguo/code-repos/digidocs-mgt/docker-compose.yml)
  - 启动 `postgres`、`backend-go`、`backend-py-worker`、`frontend`
- 根目录 [.env.example](/Users/liguo/code-repos/digidocs-mgt/.env.example)
  - 本地开发模板
- 根目录 [.env.production.example](/Users/liguo/code-repos/digidocs-mgt/.env.production.example)
  - 部署模板
- 前端镜像 [frontend/Dockerfile](/Users/liguo/code-repos/digidocs-mgt/frontend/Dockerfile)
- 前端反向代理 [frontend/nginx/default.conf](/Users/liguo/code-repos/digidocs-mgt/frontend/nginx/default.conf)

## 3. 启动方式

1. 复制生产模板：

```bash
cp .env.production.example .env
```

2. 按实际环境修改 `.env`

重点字段：

- `DATABASE_URL`
- `JWT_SECRET`
- `WORKER_CALLBACK_TOKEN`
- `CALLBACK_TOKEN`
- `OPENCLAW_BASE_URL`
- `OPENCLAW_API_KEY`
- `STORAGE_BACKEND`
- `SYNOLOGY_*`

要求：

- `CALLBACK_TOKEN` 应与 `WORKER_CALLBACK_TOKEN` 保持一致；
- `CALLBACK_BASE_URL` 在 compose 网络内应保持为 `http://backend-go:8080`；
- 若 OpenClaw 运行在宿主机且 Worker 在容器内，macOS/大多数 Docker Desktop 场景可用 `http://host.docker.internal:8001`。

3. 启动服务：

```bash
docker compose --profile app up -d --build
```

4. 查看状态：

```bash
docker compose ps
make status
```

## 4. 访问入口

- 前端：`http://localhost:${FRONTEND_PORT}`
- Go API：`http://localhost:${BACKEND_GO_PORT}/healthz`
- PostgreSQL：`localhost:${POSTGRES_PORT}`

## 5. 基础验证

建议至少执行：

```bash
make verify
make smoke
```

如果是首次部署到新环境，再补一轮人工烟测：

1. 登录
2. 创建文档
3. 上传新版本
4. 触发摘要
5. 查看 Assistant 问答结果
6. 确认/忽略建议

## 6. 当前已知限制

- `pdf`、图片 OCR、扫描件正文抽取尚未接入；
- 前端构建仍有 bundle size warning，但不阻塞部署；
- OpenClaw HTTP 端点是否启用，仍取决于目标机器上的 Gateway 配置。
